## Context

当前系统已经具备基础任务数据模型和 CRUD 能力，但任务管理仍以单一列表/编辑为主，无法满足项目推进中常见的运营场景：看板推进、排期跟踪、按项目和优先级查看、汇总健康指标和风险识别。

参考截图体现的是同一任务集的多视图运营工作台，而不是单页面报表。这个变化会同时影响前端路由结构、页面状态管理、任务查询方式和聚合展示逻辑。

## Goals / Non-Goals

**Goals:**
- 提供一个统一的任务管理工作台，支持列表、状态看板、甘特图、时间线、按项目和按优先级等视图切换。
- 在不同视图之间共享同一组筛选、搜索、分组和排序条件。
- 提供汇总指标卡片、里程碑/截止日期提示、来源类型和风险提示。
- 保留现有任务 CRUD 能力，不破坏已存在的基础编辑流程。

**Non-Goals:**
- 不在本次变更中引入任务自动调度或自动改期。
- 不实现拖拽式跨列重排作为必需能力。
- 不重做底层 work-item 数据模型，只在现有字段基础上扩展展示与查询。
- 不把 GitLab 双向同步作为本次交付的主目标。

## Decisions

### 1. 采用单一任务工作台入口，内部用视图切换承载不同表达

决定把任务管理集中到一个工作台页面或路由下，由顶部视图标签切换列表、看板、甘特图和时间线。

**理由:**
- 这是参考界面的核心特征，同一任务集需要被多个角色从不同角度阅读。
- 共享筛选和分组状态比多个孤立页面更容易保持一致。
- 便于后续扩展保存视图、收藏筛选和分享链接。

**Alternatives considered:**
- 独立页面拆分成 `/tasks/list`、`/tasks/board`、`/tasks/timeline`。
- 放在项目详情页内作为子标签页。

**Why not:**
- 独立页面会复制筛选逻辑，维护成本高。
- 放在项目详情页会限制“全局任务运营”视角，和截图中的信息架构不一致。

### 2. 使用 URL 查询参数承载可共享状态

决定将视图、筛选、分组和排序状态编码进 URL 查询参数，并在前端状态层做同步。

**理由:**
- 便于刷新后恢复、浏览器前进后退、复制链接分享。
- 能把“当前视图”变成可追踪状态，减少隐式 UI 状态。
- 对多视图工作台来说，比只放在内存 store 更稳妥。

**Alternatives considered:**
- 仅使用本地组件状态。
- 仅使用全局 store。

**Why not:**
- 仅本地状态无法分享和恢复。
- 仅全局 store 可恢复性不够直观，且 URL 不可读。

### 3. 汇总指标以“当前过滤集”为准

决定卡片指标显示当前筛选后的任务集合，而不是固定全量统计。

**理由:**
- 用户在看某个项目、某个负责人或某个状态时，应该直接看到该切片的健康度。
- 点击指标卡片可以自然落到同一过滤上下文中。

**Alternatives considered:**
- 显示全局总览并额外提供切片统计。
- 仅展示当前页统计。

**Why not:**
- 全局总览和切片视图同时存在会让用户难以判断当前筛选是否生效。
- 仅当前页统计会被分页误导，不适合健康指标。

### 4. 先用现有任务数据字段实现风险与来源展示

决定先基于现有的 source type、blocked、due date、status、updated time 等字段做风险提示，不引入新的风险引擎。

**理由:**
- 下一版本的重点是工作台表达，不是 AI 推理系统。
- 已有字段足够支持逾期、阻塞和来源标识这些基础风险信号。

**Alternatives considered:**
- 在本次变更中增加 AI 风险评分。
- 引入独立的 risk 规则引擎。

**Why not:**
- AI 和规则引擎都会增加解释成本，且会把工作台变成两个问题一起解决。
- 当前更重要的是把风险信息“可见化”。

### 5. 甘特图与时间线采用同一套任务日期语义

决定甘特图和时间线共享任务开始/结束/截止日期语义，减少两个视图的口径偏差。

**理由:**
- 同一任务在不同视图里必须表达一致，否则用户会怀疑数据正确性。
- 统一日期语义可以降低后续统计和测试复杂度。

**Alternatives considered:**
- 甘特图依赖计划开始/结束，时间线依赖实际开始/结束。

**Why not:**
- 现阶段任务数据未必完整区分计划与实际日期，容易引入口径冲突。

## Risks / Trade-offs

- [Risk] 多视图共享状态会增加前端复杂度 → [Mitigation] 用 URL 查询参数定义单一状态源，并为每个视图复用同一过滤器模型。
- [Risk] 汇总指标如果依赖全量聚合会增加后端负担 → [Mitigation] 先复用现有查询接口，必要时再补充轻量聚合端点。
- [Risk] 甘特图/时间线对日期字段质量敏感 → [Mitigation] 对缺失日期的任务降级为列表或卡片占位，不阻塞核心浏览。
- [Risk] 风险提示过多会干扰主流程 → [Mitigation] 风险标记保持非阻断、可折叠，并只突出高优先级异常。
- [Risk] 视图和筛选组合过多会让界面臃肿 → [Mitigation] 优先保留最常用的四类切片和核心筛选，其余字段收进高级筛选。

## Migration Plan

1. 先落地统一任务工作台壳层，包括路由、顶部视图切换和共享筛选状态。
2. 再接入任务列表和状态看板，保证基础任务浏览能力不回退。
3. 然后补甘特图与时间线，复用同一套任务查询结果。
4. 最后增加汇总卡片、风险提示和来源展示。
5. 如果某个图形视图存在稳定性问题，可回退到列表视图作为默认入口，同时保留其余视图。

Rollback strategy:
- 保持列表视图作为最低可用路径。
- 新视图发生问题时，只关闭对应标签入口，不影响任务 CRUD。

## Open Questions

1. 汇总指标是否需要后端聚合接口，还是可以由前端在当前过滤结果上计算？
2. 是否需要保存用户个人视图偏好，例如默认打开甘特图或按项目视图？
3. 状态看板是否必须支持拖拽改状态，还是先只做只读展示加详情编辑入口？
4. 是否需要在本次变更中把“任务”与“work item”术语统一到一套用户文案？
